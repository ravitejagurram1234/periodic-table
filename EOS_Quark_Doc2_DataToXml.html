1. Why this layer exists

QuarkXPress Server does not understand "set field NAV_DATE to 24/06/2026". It understands one thing: an XML document that describes the boxes, tables and pages of a Quark project, and what to do to them. Everything in this document is about the bridge between the engine's business data and that XML.

A run starts with raw material — rows from Oracle, literal system values, document files — and must end with a rendered PDF/JPG/QXP. In between, the engine has to answer one question for every piece of data: "which box does this go into, and what XML do I emit to put it there?" This doc traces that answer end-to-end.

Kube constraint (why it's all remote XML): the old .NET engine ran on the same host as QuarkXPress Server and could touch files directly. The Java engine runs in Kubernetes, remote from the Quark host. So every document mutation must travel as a payload over the wire — there is no local file path the engine can write. That is exactly why the serialized XML / name-value payload in this doc is the only way data reaches the document.

2. The big picture

Data flows through four stages and then forks into one of two transport payloads. Hold this diagram in mind — the rest of the doc zooms into each box.

RAW DATA FORMAT MODEL STEPS PAYLOAD & TRANSPORT ───────── ──────── ─────── ─────── ──────────────────── SQL rows ┐ BlocBox(value) ┐ UPDATE step ──> name/value pairs ──> QXPSM (SOAP) (Oracle) │ DataTypeHelper → blocsUpdate │ RunTask (all values, 1 step) Manager ├─> .outputToString() ─> "string" ─┤ ├─.getSteps() System │ (fr-FR, €, %, dates) BlocBox(srcBox) │ PAGINATION step ┐ values ┘ → blocsModify ┘ MODIFY step(s) ┴> QxpsModifier ─> XML ─> QXPS (HTTP) (structural, split @5000) Server

The key fork — and the single most misunderstood thing about this engine — is at the right edge:

What changedStep typePayloadTransportdirectCall
A field's text/value (a SQL result, a system value)UPDATEname/value pairsQXPSM — SOAPfalse
Structure (create/move/delete boxes, tables, pages)MODIFY / PAGINATIONModifier XMLQXPS — HTTPtrue (default)
Read this carefully — it is the opposite of the intuitive framing. Plain value/text fills go over SOAP (QXPSM); heavyweight structural edits go over HTTP (QXPS). The decision is made once, in RunTask.getSteps(), by setting step.setDirectCall(false) on the value step and leaving it true on the structural steps. QxpsCallerBusiness.executeStep() then branches: if (step.isDirectCall()) executeDirectCall(/*HTTP*/) else executeSoapCall(/*SOAP*/).

3. Stage 1 — Acquire the data & decide its target box

Two task types produce ordinary field values. Each one's *ProcessStrategy takes a value, formats it, wraps it in a BlocBox, and drops it into the task's blocsUpdate map keyed by bloc name (the box name in the Quark template).

SYSTEM tasks — a single literal value

SystemTaskProcessStrategy has two modes:

  • VALUE mode — format task.getValue() and store it as new BlocBox(task, destinationBlocName, formattedValue) in task.getBlocsUpdate(). This is a pure text change → ends up on the SOAP path.
  • BOX mode — the change is a box (a template element with geometry), not just text. It builds BlocBox from a srcBox/srcExtraBox SOAP Box and routes it to blocsModify → structural/HTTP path.

SQL tasks — one row → one box

SqlTaskProcessStrategy delegates to ProcessSqlBusiness.addBlocs(). The query is expected to return rows with these well-known columns:

SQL columnBecomesRole
NOM_BLOCBlocBox.nameWhich box in the template receives the value
VALEUR_BLOCBlocBox.value (after formatting)The text to put in the box
DESCRIPTION_BLOC / INFO_BLOCDataNameValue metadataCarried only into the store-data stream (§12), not the document

For each row: blocName = row[NOM_BLOC]; the value is formatted (Stage 2); a BlocBox is created with action = BlocActionEnum.UPDATE and added to task.getBlocsUpdate(). So one SQL row produces one named value fill.

Other task types (Document, QXP_Previous, Dynamique, Compartiment) build structural blocs (boxes/tables/pages with geometry) into blocsModify — those are covered as the structural path below and in Doc 1. This section focuses on the value-producing path because that is where "data → text → XML" is most concrete.

4. Stage 2 — Format the text (DataTypeHelper)

Before a value can become XML it becomes a string, formatted exactly as it should appear in the rendered document. This is the job of DataTypeHelper.outputToString(value, dataType, nbDecimal, showZero, nullString, decimalSignificative) — the single chokepoint both SYSTEM and SQL strategies call.

DataTypeTransformationExample out
INTParse to integer, group with fr-FR thousands separator1 234 567
DECIMALBigDecimal to nbDecimal places, HALF_UP rounding, fr-FR grouping; decimalSignificative toggles fixed trailing zeros vs suppression1 234,50
CURRENCYDecimal core + " €" suffix (fr-FR symbol). No ×100.1 234,50 €
POURCENTAGEDecimal core + single " %" suffix. No ×100 — values are already stored scaled (15 = 15%).15,00 %
DATELocalDate formatted dd/MM/yyyy24/06/2026
DATE_TIMELocalDateTime formatted dd/MM/yyyy HH:mm:ss24/06/2026 14:30:00
otherRaw toString()unchanged

showZero=false turns a zero into nullString (often blank); nullString substitutes for missing values. The result of this stage is a plain String — that string is what eventually appears as text content inside the XML.

Two documented behaviours worth knowing (from the verified fix batches):
• POURCENTAGE emits a single " %" with no ×100. The legacy .NET engine path multiplied by 100 and doubled the suffix (rendering 15 as "1 500,00 % %" — an author-flagged bug). The user approved deviating to the correct output, which matches .NET's other formatter (QXP_Format_Helper).
• fr-FR grouping char: JDK 17 / CLDR may use a narrow no-break space (U+202F) where .NET used U+00A0. This affects all grouped numbers and should be checked against a golden file.

5. Stage 3 — Attach to the model (BlocBox → task maps)

The formatted value is now wrapped in a BlocBox and parked on the task in one of two maps. Which map decides which transport.

Map on TaskBaseHoldsKey carried by BlocBoxDestination
blocsUpdate : Map<String,BlocBox>Text/value changesname + value (String)SOAP value path
blocsModify : Map<String,BlocBase>Structural changesname + srcBox (SOAP Box) + action + pageIdHTTP structural path

The serializer / payload builder later reads exactly these fields. For a value bloc it reads BlocBox.getValue(). For a structural bloc it reads BlocBox.getSrcBox() — a fully-populated SOAP Box whose own text value lives at srcBox.text.story.paragraphs[0].richText[0].value and whose geometry lives at srcBox.geometry.position.

BlocActionEnum on each bloc (UPDATE, CREATE, MOVE, REMOVE/NONE) is what the serializer turns into the XML OPERATION attribute on the resulting element.

6. Stage 4 — Split into steps (RunTask.getSteps())

All of a run's blocs are reorganised into an ordered list of RunTaskSteps. This is where the SOAP-vs-HTTP decision is physically made by flipping directCall.

RunTask.getSteps() → ordered List<RunTaskStep> [0] UPDATE step setDirectCall(false) ← ALL value blocs, never split — exactly ONE step [1] PAGINATION step directCall = true ← page create/remove blocs [2..] MODIFY step(s) directCall = true ← structural blocs, split every `splitStepBoxNumber` (default 5000) boxes
  • The UPDATE step collects every value bloc from every task into a single step (it is never batched), and marks it directCall=false → it will go over SOAP.
  • PAGINATION and MODIFY steps keep the default directCall=true → HTTP. Big structural docs are chunked at splitStepBoxNumber so no single call is too large.

Consequence: a SQL/System-only fact-sheet (no structural changes) produces exactly one SOAP step and zero HTTP modify calls. A heavy Dynamique/Compartiment doc produces several HTTP modify steps.

Per-step the engine also resolves each box's page & layout via TaskBase.evaluateInfo() → QxpXml.getPageNum/getLayoutName, re-reading the latest saved document XML (each step saves a new version). A capacity guard (RunTaskStep.evaluateLimitation, against nb-box-max / average-box-size) only matters for very large structural docs.

7. The two payloads (the fork in detail)

QxpsCallerBusiness.executeStep(step, ...) builds a QxpsModifier from the step's structural blocs and then branches on step.isDirectCall():

QxpsModifier modifier = new QxpsModifier();
modifier.addRange(step.getBlocsModify());

if (step.isDirectCall()) {
    executeDirectCall(run, step, modifier, ...);   // HTTP / QXPS
} else {
    executeSoapCall(step, modifier, ...);           // SOAP / QXPSM
}

7a. The value payload — name/value pairs SOAP

For the UPDATE step, RunTaskStep.evaluateModifierAndValues() walks blocsUpdate and emits a flat list of NameValueParam(paramName, textValue) — box name → formatted string:

for (BlocBase bloc : blocsUpdate) {
    if (bloc instanceof BlocBox b) {
        NameValueParam nv = new NameValueParam();
        nv.setParamName(b.getName());     // box name in the template
        nv.setTextValue(b.getValue());    // the formatted string from Stage 2
        nameValues.add(nv);
    }
}

That list is handed to QxpsmSoapClient.executeStep(documentName, nameValues, project /*usually null for UPDATE*/, saveAsPath, newGabaritName). The SOAP side carries the pairs as method parameters; QuarkXPress Manager finds each named box and replaces its text. No box geometry is sent — the box already exists in the template, we are only refilling it.

There are two serialized forms of name/value in the codebase: NameValueSerializer emits a compact <PARAMSVALUE><NAMEVALUEPARAM PARAMNAME="…" TEXTVALUE="…"/>…</PARAMSVALUE> XML; ParamsValueMessage emits the HTTP-GET query-string form name1=value1&name2=value2 used when value updates ride along on a combined HTTP call.

7b. The structural payload — Modifier XML HTTP

For MODIFY/PAGINATION steps, the structural blocs are folded into the Modifier object tree and serialized to XML. QxpsModifier.add(bloc) routes each bloc by type and walks/creates the nesting Projet → Layout → Spread → {Box | Table | Page | Group}:

private void addBlocBox(BlocBox bloc) {
    ModifierSpread spread = addGetSpread(bloc);     // find/create layout+spread
    ModifierBox box = new ModifierBox();
    box.setName(bloc.getName());
    box.setAction(bloc.getAction());                // → OPERATION attribute
    box.setPageId(bloc.getPageId());                // → GEOMETRY/@PAGE
    box.setValue(bloc.getValue());
    box.setSrcBox(getBox(bloc, false));             // the SOAP Box (text+geometry+style)
    spread.getBoxes().put(box.getName(), box);
}

Then modifier.getProject() converts the whole tree into a SOAP Project (ModifierProjet.getSdkProject() → ModifierLayout.getSdkSpreads() → ModifierSpread.getSdkBoxes()/getSdkTables()/getSdkPages(), applying OPERATION = CREATE/MOVE/DELETE/NONE per bloc action). Finally QxpsProjectSerializer.toBytes(project) streams it to XML bytes.

On the HTTP side that XML is uploaded to the document pool as its own file and then referenced by name in one combined call (Kube can't hand the server a local file):

byte[] modifyXml = QxpsProjectSerializer.toBytes(modifier.getProject());
String modifyFileName = run.getRunProperties().getPoolPath(/* MODIFY_<timestamp> */);
qxpsHttpClient.execute(modifyFileName, new AddFileMessage(modifyXml));   // 1) upload
messages.add(new ModifyMessage(modifyFileName));                         // 2) reference
messages.add(new ParamsValueMessage(nvArray));   // any inline value updates
messages.add(new SaveAsMessage(saveAsPath, newGabaritName, true, false));
messages.add(new QxpRenderMessage());
qxpsHttpClient.executeCombined(documentName, messages);                  // 3) ONE call

8. The Modifier XML schema

This is the actual document grammar QuarkXPress Server consumes. All element and attribute names are UPPERCASE on the wire (verified directly against the serializer's writeStartElement("…") / writeAttr(w,"…") literals — note the .NET source uses lowercase C# property names but emits the same uppercase XML). The serializer omits any attribute or element whose value is null/empty, so real payloads are much sparser than the full grammar below.

<PROJECT>
  <LAYOUT NAME="…" UID="…">
    <SPREAD OPERATION="CREATE|…" UID="1">

      <PAGE OPERATION="CREATE|DELETE" MASTER="…" POSITION="…" UID="2">
        <ID NAME="Page2" UID=""/>
      </PAGE>

      <BOX OPERATION="CREATE|MOVE|NONE" BOXTYPE="CT_TEXT|CT_PICT"
           COLOR="" SHADE="" OPACITY="" ANCHOREDIN="">
        <ID NAME="NAV_DATE" UID=""/>
        <TEXT VERTICALALIGNMENT="…" COLUMNS="…" …>
          <INSET TOP="" BOTTOM="" LEFT="" RIGHT="" ALLEDGES=""/>
          <STORY CLEAROLDTEXT="true" FITTEXTTOBOX="" CONVERTQUOTES="" INCLUDESTYLESHEETS="">
            <PARAGRAPH PARASTYLE="" PARACHAR="" MERGE="">
              <FORMAT ALIGNMENT="" LEADING="" SPACEBEFORE="" SPACEAFTER="" …/>
              <RICHTEXT FONT="" SIZE="" BOLD="" ITALIC="" COLOR="" …>24/06/2026</RICHTEXT>
            </PARAGRAPH>
          </STORY>
        </TEXT>
        <PICTURE FIT="" SCALEACROSS="" OFFSETACROSS="" …/>      <!-- CT_PICT boxes -->
        <GEOMETRY SHAPE="" PAGE="1" ANGLE="" LAYER="">
          <POSITION><TOP>100</TOP><LEFT>50</LEFT><BOTTOM>200</BOTTOM><RIGHT>150</RIGHT></POSITION>
          <RUNAROUND TYPE="" …/>
          <SPLINESHAPE …><CONTOURS><CONTOUR><VERTICES><VERTEX>
            <LEFTCONTROLPOINT X="" Y=""/><VERTEXPOINT X="" Y="" TAG=""/><RIGHTCONTROLPOINT X="" Y=""/>
          </VERTEX></VERTICES></CONTOUR></CONTOURS></SPLINESHAPE>
        </GEOMETRY>
        <CONTENT CONVERTQUOTES="" INCLUDESTYLESHEETS="" FONTNAME="">…</CONTENT>
        <SHADOW COLOR="" ANGLE="" DISTANCE="" …/>
        <FRAME STYLE="" WIDTH="" COLOR="" …/>
      </BOX>

      <TABLE OPERATION="CREATE|DELETE" COLUMNCOUNT="" ROWCOUNT="" MAINTAINGEOMETRY="" AUTOFIT="" …>
        <ID NAME="HOLDINGS" UID=""/>
        <ADDCELLS TYPE="ROW" BASEINDEX="" INSERTCOUNT="" INSERTPOSITION="" KEEPATTRIBUTE=""/>
        <DELETECELLS TYPE="ROW" BASEINDEX="0" DELETECOUNT="1"/>      <!-- ← ModifierLigne -->
        <COLSPEC><COLUMN COLUMNWIDTH="" COLOR="" …><GRIDLINE STYLE="" WIDTH="" …/></COLUMN></COLSPEC>
        <ROW ROWHEIGHT="" COLOR="" …>
          <CELL BOXTYPE="" COLOR="" …>
            <CONTENT …/>
            <TEXT><STORY CLEAROLDTEXT="true"><PARAGRAPH><RICHTEXT>cell text</RICHTEXT></PARAGRAPH></STORY></TEXT>
          </CELL>
          <GRIDLINE …/>
        </ROW>
        <TABLEBREAK/><FRAME …/><GEOMETRY …/><SHADOW …/>
      </TABLE>

    </SPREAD>
  </LAYOUT>
</PROJECT>

Notice the three places your data actually lands:

  • Text value → BOX/TEXT/STORY/PARAGRAPH/RICHTEXT text node (and the cell equivalent inside TABLE/ROW/CELL).
  • Which box → BOX/ID/@NAME (matches the NOM_BLOC / destination bloc name).
  • Where on the page → BOX/GEOMETRY/@PAGE + POSITION/{TOP,LEFT,BOTTOM,RIGHT}.

STORY/@CLEAROLDTEXT="true" is what makes a fill replace (not append to) the box's existing text. Row deletions are expressed as DELETECELLS children built from ModifierLigne entries; a whole table marked REMOVE becomes TABLE/@OPERATION="DELETE". A Group has no element of its own — its member boxes are unpacked into the spread as BOX elements with OPERATION="CREATE".

9. The serializer itself (QxpsProjectSerializer)

A streaming StAX writer — infra/interop/qxps/helper/QxpsProjectSerializer.java. Entry points: toXml(Project) → UTF-8 String, toBytes(Project) → bytes for the HTTP body. It walks the SOAP Project tree with one writer method per node: writeProject → writeLayout → writeSpread → {writePage, writeBox, writeTable} and downward into text/geometry.

Rule (helper)Behaviour
writeAttr(w,name,value)Writes the attribute only if value is non-null and non-empty
writeElem(w,name,value)Writes a simple text element only if non-null/non-empty
writeElemSkipFalse(...)Skips "false" flags (e.g. SUPPRESSOUTPUT) — server default is false

This "omit empties" policy is why the on-the-wire XML is far smaller than the full grammar — every box only carries the handful of attributes that differ from template defaults.

10. The in-memory element model (TElement)

Structural blocs don't carry loose strings — they carry TElement objects (the engine's representation of a template box/table/group), which wrap a SOAP element and add geometry. These feed the Modifier* classes.

ClassWrapsCarries
TElement (abstract)—name, geometry left/top/right/bottom, derived width/height, evaluated flag
TBoxSOAP Box (+ optional srcExtraBox)type (CT_TEXT/CT_PICT/…), page; evaluate() reads srcBox.geometry.position into BigDecimals and computes size
TTableSOAP Tablegeometry evaluation is a placeholder (no-op)
TGroupSOAP Groupresolves member boxes by name, computes a bounding box; nested sub-groups throw ExceptionTElement

TElementHelper is the toolbox that mutates these: updateNameValue(tBox, name, value) sets srcBox.name and the rich-text value; updatePosition(tBox, cell) writes the POSITION coordinates; deep-clone helpers (cloneTBox/cloneTGroup/cloneTTable via in-memory serialization) give each repeated element an independent copy. It also holds the static template library — EMPTY_BLOC, IMG, PDF_1, PDF_N, RTF_DOC_XTG, MOVE_BLOC, EMPTY_TABLE — the prototype boxes Document/Dynamique tasks clone and fill.

11. The mirror image — reading XML back (QxpXml)

Serialization is only half the conversation. After each step the engine downloads the document's current XML and parses it to learn where things ended up. QxpXml (domain/xml/QxpXml.java) is the read side — pure XPath queries over the returned document:

MethodXPath gistUsed for
getValue(blocName)//ID[@NAME='…']/..//RICHTEXT/text()Read a box's current text (e.g. differential store)
getPageNum(blocName)…/GEOMETRY/@PAGEWhich page a box landed on (per-step evaluateInfo)
getLayoutName(blocName)/PROJECT/LAYOUT[//ID[@NAME='…']]/ID/@NAMEWhich layout owns the box
getBlocInfo(blocName)name+page+UID+POSITION bundleAnchors / INCORPORATE geometry
getNamesValuesBoxes()all CT_TEXT ID/@NAME + valuesDifferential document store (§12)
Write vs read in one line: the Modifier path takes in-memory blocs and produces XML to send to the server; QxpXml takes the XML the server returns and extracts document state back into the engine. Same grammar, opposite direction.

12. The third stream — store-data (historisation)

Not all data goes to Quark. When store_data is enabled, values are also captured into a separate audit stream for traceability — and the engine keeps two different drawers that legitimately differ:

DrawerSourceWhat it captures
SQL datacollectSqlData from run.sqlDataNamesValues (simple, diff=false)The raw query outputs as the engine computed them
DOCUMENT datacollectDocumentData after refreshGabaritXml (differential, diff=true)What is actually in the finished boxes — incl. non-SQL values & formatting, read back via QxpXml.getNamesValuesBoxes()

The store stream is gated by a bitwise flag: StoreDataType.hasFlag(run.getRunProperties().getStoreDataTypeCode(), StoreDataType.SQL | StoreDataType.DOCUMENT) — the code 0x03 enables both. Captured rows are DataNameValue(name, value, description, info) on task.dataNameValues, persisted via the data-storage DAO; they are never sent in the Quark transport call.

This is why the DESCRIPTION_BLOC/INFO_BLOC SQL columns exist — they ride only into this audit drawer, not into the document XML.

13. Worked example — one SQL value, end to end

A Plaquette fact-sheet has a SQL task returning a row that fills the box NAV_DATE with a date.

1 · Acquire. ProcessSqlBusiness.addBlocs() reads the row: NOM_BLOC = "NAV_DATE", VALEUR_BLOC = 2026-06-24, task dataType = DATE.
2 · Format. DataTypeHelper.outputToString(2026-06-24, DATE, …) → "24/06/2026".
3 · Attach. new BlocBox(task, "NAV_DATE", "24/06/2026"), action=UPDATE → task.blocsUpdate["NAV_DATE"]. (If store-data on: also DataNameValue("NAV_DATE","24/06/2026",…).)
4 · Step. RunTask.getSteps() puts it in the single UPDATE step, directCall=false.
5 · Payload. evaluateModifierAndValues() → NameValueParam("NAV_DATE","24/06/2026").
6 · Transport. executeSoapCall → QxpsmSoapClient.executeStep(doc, [NAV_DATE=24/06/2026], project=null, saveAs…). QuarkXPress Manager finds the box named NAV_DATE and sets its text. No geometry sent.

Now contrast a structural example — INCORPORATE adds a new text box at a computed position:

3' · Attach. bloc goes to blocsModify carrying a SOAP Box (text + geometry.position).
5' · Payload. QxpsModifier nests it under Projet→Layout→Spread→Box; QxpsProjectSerializer.toBytes() emits:
<PROJECT><LAYOUT NAME="L1"><SPREAD UID="1">
  <BOX OPERATION="CREATE" BOXTYPE="CT_TEXT">
    <ID NAME="INC_42" UID=""/>
    <TEXT><STORY CLEAROLDTEXT="true"><PARAGRAPH><RICHTEXT>…</RICHTEXT></PARAGRAPH></STORY></TEXT>
    <GEOMETRY PAGE="3"><POSITION><TOP>120</TOP><LEFT>40</LEFT><BOTTOM>180</BOTTOM><RIGHT>200</RIGHT></POSITION></GEOMETRY>
  </BOX>
</SPREAD></LAYOUT></PROJECT>
6' · Transport. XML uploaded to pool as MODIFY_<ts>, referenced by ModifyMessage, sent with SaveAs+Render in one combined HTTP call to QXPS.

14. .NET parity & deliberate deviations

The Java serializer is a faithful port of QXP.Engine.Core/Business/QXP_Project_Serializer.cs (its Add_Project → Add_Layout → Add_Spread → Add_Box/Add_Table → … → Add_RichText/Add_Geometry/Add_Position methods produce the identical element tree). The Modifier object model maps 1:1: .NET QXPS_Modifier → Projet → Layout → Spread → Box/Table/Page/Group ↔ Java QxpsModifier → ModifierProjet → ModifierLayout → ModifierSpread → ModifierBox/ModifierTable/ModifierPage/ModifierGroup. Transport messages match: QXPS_Message_Modify (structural XML file), QXPS_Message_ParamsValue (name=value query), QXPS_Message_Xml (read).
